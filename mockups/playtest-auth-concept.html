Architecture — what actually ships

1. SQL migration

Run once in Supabase SQL Editor. Adds the is_playtest flag, doesn't touch existing rows.

alter table public.profiles
  add column if not exists
  is_playtest boolean default false;

2. Seed 5 accounts (manual, Supabase dashboard)

Authentication → Users → "Add user" 5×. Set password Playtest2026! + email_confirmed_at = now() so email confirmation is bypassed.

3. Mark them as playtesters + Pro

One SQL block — flips is_playtest + adds Pro subscription rows. 9999-12-31 end-date means is_pro() stays true forever.

update public.profiles
set is_playtest = true
where email like '%@certanvil-playtest.com';

insert into public.subscriptions
  (user_id, tier, status, current_period_end)
select id, 'pro', 'active', '9999-12-31'
from auth.users
where email like '%@certanvil-playtest.com'
on conflict (user_id) do update set
  tier = 'pro',
  status = 'active',
  current_period_end = '9999-12-31';

4. Auth modal — read URL param + show password mode

Add ~30 LOC to landing/auth.js: detect ?auth=password on page load, swap the auth modal's default mode from magic-link to password. Standard Supabase signInWithPassword({ email, password }) handles the rest.

// landing/auth.js (excerpt)
const isPlaytestMode =
  new URLSearchParams(location.search)
    .get('auth') === 'password';

if (isPlaytestMode) {
  authModal.setMode('password');
  // renders password fields + tester pill
}

Why this design (vs alternatives)

What this gets right

  • Zero new public UI surface — public users never see "Sign in with password"
  • Standard Supabase password auth — no custom JWT minting or auth bypass
  • Re-login is free — testers can clear cookies, sign back in with same creds
  • 2FA bypass is automatic — these accounts simply don't have 2FA configured
  • Cookie-based cross-subdomain session via Phase C′ — sign in once on certanvil.com, all *.certanvil.com see them
  • Analytics isolation via is_playtest — testers can be filtered out of revenue dashboards forever
  • Admin role stays exclusive to you — testers get Pro via subscriptions, not admin
  • Reversible: 1 SQL deletes everything when playtest ends

Trade-offs accepted

  • If a tester loses the URL, they're stuck until you re-text it — solved by "bookmark this URL" foot copy
  • Shared password — if 1 tester leaks Playtest2026!, all 5 accounts compromised. Mitigation: each has unique email, can SQL-revoke 1 without affecting others
  • Tester emails are at a fake domain — they won't actually receive a magic link if they try the magic-link flow by accident. Mitigation: tell them in the onboarding card to use the special URL only
  • You don't get a fancy admin dashboard — SQL-query via Supabase editor is the workflow
