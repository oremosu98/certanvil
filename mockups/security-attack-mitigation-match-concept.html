Why parameterised queries beat WAF rules here

A WAF rule can detect-and-block obvious injection patterns at the edge — but it's a signature game (attackers fuzz around it). Parameterised queries remove the vulnerability entirely: user input is bound as data, never executed as SQL syntax. CompTIA prefers the answer that prevents the class of attack, not the one that catches some of it.

Implementation notes scope · schema · files

Drill mechanics

  • Pool size: ~96 attack/mitigation pairs across 5 categories (~18-20 per cat).
  • Question format: stem = attack name + sub-line + objective tag · 4 MCQ options (1 correct + 3 plausible distractors). All 4 options are real mitigations — the trap is picking a less-correct one.
  • Distractor authoring rule: every distractor must be a mitigation that exists for SOMETHING in the pool — picking it isn't unreasonable, it's just less-correct for this attack.
  • Scoring: per-pair mastery (caps at 3, reset on wrong). Same algorithm as Acronym Blitz STORAGE.SAB_MASTERY.
  • Session length: 10 questions weighted toward weakest category + recently-wrong pairs.
