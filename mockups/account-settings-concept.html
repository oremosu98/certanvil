State 1 design notes:
  • Identity card at top establishes "this is you" before any controls. Tier pill colour-codes free vs paid.
  • §02 Subscription shows BOTH active certs (Network+ unlocked free) AND locked certs (Security+ requires upgrade). Single page covers entitlement state. Per-cert "Open →" goes to the cert app via /?cert=<id> (the v4.89.5 URL handler).
  • §03 Security shows the magic-link method clearly. Linked accounts row is visible but italic-deferred — sets the OAuth-readiness expectation without coding it now (decision #3).
  • §04 Notifications uses 4 toggles to start. Each has a single sentence explaining the trigger so the user can decide. Marketing OFF by default per user-respect default.
  • §05 Data export is GDPR-friendly + a quiet trust signal. JSON not CSV (preserves nesting + types).
  • §06 Danger zone visually distinct (red-tinted background + border). Mirrors the cert app's existing Settings page Danger zone so the pattern is consistent across surfaces.
  • NO admin sections appear here. Identical for every signed-in user — that's the whole point of decision #2.

State 4 design notes:
  • Topbar nav swaps for admin context — primary site nav (Why / Certs / Pricing) replaced with admin sections (Dashboard / Users / Certs / Operations / Telemetry). Logo gets "· ADMIN" suffix.
  • 4-stat hero grid at top for at-a-glance health: total users, weekly actives, cert packs live, current app version. Reads like a deploy status board.
  • Users table is the bread-and-butter — currently 1 row (you), but scales to N. Includes inline role / tier / cert columns + per-row View action that opens user-detail panel (drill-down via Phase G).
  • Cert packs table centralizes visibility decisions — toggle Network+ (public) vs Security+ (private) vs coming-soon tiles without code changes. Each row's Configure action opens cert-pack settings.
  • Recent ships table turns the existing CHANGELOG.md into a queryable view: last 10 versions with UAT pass count, deploy outcome, time. View on GitHub links to the full commit history.
  • Telemetry section is dashed-border placeholder — visible to anchor the spot but explicitly marked as deferred. Sets expectation, doesn't pretend the data's there.
  • RLS-gated on the server: profiles.role='admin' via the existing is_admin() SECURITY DEFINER helper from Phase C′. Without admin role, the page returns 403 — not just visually hidden.

scoping for next ship

  • SUPABASE — RLS policies likely need touching on profiles for the email-change flow + RPC for account deletion. Existing is_admin() helper covers /admin gating.
Phase ordering recommendation:
  • Ship 1: Account page (states 1 + 3) — Profile, Security (read-only), Data export, Sign out, Danger zone. Skip subscription details (placeholder badge only, no Stripe). Skip notifications (toggles render but don't wire).
  • Ship 2: Notifications — wire the 4 toggles to profiles.metadata.notifications.* + a server-side cron + Resend email integration (already configured for the notify modal).
  • Ship 3: Admin dashboard (state 4) — users table + cert pack visibility toggle + recent ships pull from GitHub API.
  • Ship 4 (Phase G): Subscription + billing — Stripe portal integration, cert entitlements gating, paid tier upgrade flow. State 2 lights up.
Decisions still open (push back if needed):
  • "Marketing emails" toggle defaults — OFF respects user, ON gives reach. Recommendation: OFF default with friendly nudge during sign-up.
  • Account deletion — soft-delete (mark deleted, retain for 30d) vs hard-delete (immediate purge). Recommendation: 7-day soft-delete + immediate UI confirmation, hard-purge after window.
  • Email-change flow — verification email to NEW address, link confirms switch. Standard pattern; needs a Supabase RPC.
  • Should cert app /settings redirect to landing /account for the truly-cross-cert items (email, billing) and KEEP only app-prefs (exam date, daily goal, API key)? Recommendation YES — it removes confusion.
